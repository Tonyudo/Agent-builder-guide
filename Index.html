Foundations

What is an AI Agent?

An agent is an AI system that can reason, plan, and take actions autonomously to complete a goal — not just answer a question.

The key difference: LLM vs Agent

LLM (Chatbot)Agent
Responds to one promptExecutes multi-step tasks
No memory between turnsCan maintain context and state
Text in, text outCan call APIs, browse, write files
Single passLoops until goal is met

The Agent Loop

At the core of every agent is a loop: perceive → think → act → observe → repeat.

1
Perceive
The agent receives input: a user message, a file, a scheduled trigger, a webhook.
2
Think / Plan
The LLM reasons about what to do next: call a tool? ask a clarifying question? finish?
3
Act
The agent calls a tool, writes to a database, sends an email, or returns a response.
4
Observe
The result of the action is fed back into context. The agent decides whether to loop or stop.

When to Build an Agent

Not every problem needs an agent. Here's how to decide.

✅
Good candidates for an agent:
  • Tasks with multiple steps that depend on each other
  • Tasks that require browsing, reading, or writing external data
  • Workflows where the path isn't fully known upfront
  • Repetitive knowledge work: research, summarization, drafting at scale
⚠️
Think twice before building an agent if:
  • A simple prompt or template already solves the problem
  • The task is purely deterministic (use regular code)
  • Errors in the output have severe, hard-to-reverse consequences
  • Latency and cost are critical constraints

Core Concepts

The 6 building blocks you must understand before writing a single line of agent logic.

1. The LLM (The Brain)

The large language model is the reasoning core of your agent. It reads the system prompt + conversation history + tool results, and decides what to do next. Claude and GPT-4 are both strong choices. Claude excels at long-context reasoning and following complex instructions reliably.

2. The System Prompt

This is the agent's permanent instructions. It defines: who the agent is, what it can and cannot do, how it should behave, what tools it has access to. Treat your system prompt like a job description — be specific.

3. Tools (Actions)

Tools are functions the LLM can call to interact with the outside world: search the web, read a file, call an API, run code. Without tools, an agent can only produce text. With tools, it can take real action.

4. Memory

LLMs are stateless by default — they forget everything after each API call. Agents use different memory strategies: in-context (conversation history), external (database lookup), and episodic (summaries of past sessions).

5. Context Window

The context window is how much text the LLM can 'see' at once. Claude's context window is up to 200K tokens (~150K words). Long-running agents need strategies to avoid overflowing it: summarization, pruning, retrieval.

6. Guardrails

Agents can make mistakes, get stuck in loops, or take unintended actions. Guardrails are safety mechanisms: max iteration limits, human-in-the-loop checkpoints, output validation, rate limiting on destructive actions.

Build Your First Agent

A practical walkthrough: building a research assistant agent using Claude.

1
Define the job to be done
Write one sentence: 'This agent will ___.' Be specific. Vague goals create vague agents.
2
Write the system prompt
Define persona, scope, output format, constraints. Start with 200–400 words. Test, then expand.
3
Choose your tools
For a research agent: web search + read URL + write file. Add only what's needed.
4
Define the loop & stopping condition
When does the agent stop? After N steps? When it produces a specific output? Always define this before you build.
5
Test with edge cases
What happens when a tool fails? When input is ambiguous? Build for the happy path, then test the unhappy paths.

Example: Minimal Research Agent (Claude API)

// System prompt for a research agent
const systemPrompt = `You are a research assistant.
Given a company name, you will:
1. Search for recent news about the company
2. Find their main product and competitors
3. Return a structured brief with: summary, key facts, 3 competitors

Use the search_web tool to gather information.
Stop when you have enough to complete the brief.
Be concise. Output in JSON.`;

// Tool definition
const tools = [{
  name: "search_web",
  description: "Search the internet for current information",
  input_schema: {
    type: "object",
    properties: {
      query: { type: "string", description: "Search query" }
    },
    required: ["query"]
  }
}];
💡
Start with claude-sonnet for development (fast + cheap), switch to claude-opus for production where quality matters most.

Prompting Basics for Agents

Agent prompting is different from chatbot prompting. Here's what changes.

System Prompt Structure

A strong agent system prompt has 5 parts:

1
Identity
Who is the agent? What is its role?
2
Goal
What is the agent trying to accomplish?
3
Tools available
List each tool and when to use it.
4
Constraints
What can the agent NOT do? Hard rules.
5
Output format
Exactly how should the final answer look?

Prompting Patterns That Work

✅
Chain of thought: Add "Think step by step before acting" to get more reliable planning.
✅
Explicit stopping: Tell the agent exactly when to stop. "Stop after 5 searches" beats "stop when you have enough."
⚠️
Avoid vague permissions: "Do whatever it takes" will bite you. Always scope the agent's authority.
⚠️
Don't over-constrain: Too many rules create contradictions. Keep it to 5–8 core rules.

Tools & Functions

Tools transform an agent from a text generator into a system that can act. Here's how to design them well.

Tool Design Principles

  • One tool, one job. Don't build a 'do_everything' tool.
  • Name tools clearly: search_web, read_file, send_email.
  • Write descriptions as if explaining to a junior employee.
  • Always return structured errors — never let a tool silently fail.
  • Make destructive tools (delete, send, post) require explicit confirmation.

Common Tool Categories

CategoryExamplesRisk level
Read-onlysearch_web, read_file, get_calendarLow
Write / Createcreate_doc, write_file, add_rowMedium
Send / Publishsend_email, post_slack, publish_pageHigh
Executerun_code, deploy, delete_recordVery High
💡
For Claude: tool descriptions are part of the system prompt. Spend as much time on tool descriptions as on the rest of your prompt.

Memory & Context

How to give your agent a reliable memory without blowing the context window.

The 4 Types of Agent Memory

📜
In-Context
The full conversation history. Simple, but limited by token count.
🗄️
External DB
Store facts in a database; retrieve them via a tool call.
🔍
RAG (Retrieval)
Embed documents; retrieve only what's relevant to the current query.
📝
Episodic Summary
Summarize past sessions and inject the summary at the start of new ones.

Context Window Management

Rule of thumb: keep context under 70% of the model's max to avoid degraded performance.

  • Summarize tool results before adding to context
  • Use 'sliding window' — keep only the last N messages
  • Store intermediate results externally and retrieve them only when needed

Multi-Step Flows

Design reliable agent workflows that handle branching, errors, and partial results.

Flow Patterns

PatternWhen to useExample
SequentialSteps must happen in orderGather → Analyze → Draft → Send
ParallelIndependent sub-tasksSearch 3 sources simultaneously
ConditionalBranch on data or confidenceIf low confidence → ask human
Retry loopTool failures, validation errorsTry up to 3 times, then escalate
💡
Always add a maximum step count. An agent with no upper bound will keep looping on ambiguous tasks. 20–30 steps is usually enough.

Multi-Agent Systems

When one agent isn't enough. Orchestrate multiple specialized agents to tackle complex, parallelizable workflows.

Orchestrator–Subagent Pattern

One orchestrator agent decomposes the goal and delegates to specialized subagents.

✅
Good uses of multi-agent: long research tasks (parallel search + analysis), software dev (planner + coder + reviewer), content production (researcher + writer + editor).
⚠️
Multi-agent adds latency, cost, and debugging complexity. Start with a single agent. Only split when you hit a clear wall.

Trust & Permission Boundaries

Subagents should have minimum required permissions. The orchestrator should validate before acting on subagent output.

Evals & Testing

You can't improve what you don't measure. Here's how PMs should think about evaluating agents.

What to Measure

MetricWhat it tells you
Task completion rateDoes the agent finish what it starts?
Output quality scoreIs the result accurate and useful?
Steps per taskEfficiency. High step count = inefficient planning.
Cost per taskToken spend. Essential for production.
Error rateHow often does the agent fail or hallucinate?

Building an Eval Suite

1
Collect 20–50 real test cases
Mix: easy, hard, edge cases, and failure modes you've seen.
2
Define a scoring rubric
LLM-as-judge works well: use a grader prompt to score output 1–5.
3
Run evals on every prompt change
Prompt engineering without evals is guesswork.

Production & Safety

Shipping an agent to real users is different from building a demo. Here's what changes.

Production Checklist

  • Rate limiting on all destructive tool calls
  • Human-in-the-loop for high-stakes actions (send, publish, delete)
  • Hard max on iterations (prevents infinite loops)
  • Output validation before any external action
  • Full logging: every step, tool call, and result
  • Alerting on error rate spikes
  • Graceful degradation: agent should fail safely, not silently
  • Cost budget per run — alert when exceeded
🚨
Prompt injection is real. If your agent reads from user-controlled content (emails, docs, web pages), a malicious actor can embed instructions that hijack your agent. Always sanitize inputs.

PM Checklist: Ship an Agent

From idea to production. Check every box.

Discovery

  • Defined the exact job-to-be-done in one sentence
  • Validated that an agent is the right solution
  • Identified the user and their actual workflow

Design

  • System prompt written and reviewed by someone not on the team
  • Tools defined with clear names, descriptions, and error handling
  • Stopping conditions explicitly defined
  • Edge cases documented: empty input, tool failure, ambiguous request

Build & Test

  • Tested with 20+ diverse inputs
  • Eval suite created and baseline score recorded
  • Cost per task estimated and acceptable
  • Average latency tested and acceptable

Launch

  • Logging enabled for all steps and tool calls
  • Rate limits and guardrails in production
  • On-call process defined for agent failures
  • User feedback mechanism in place
  • Rollback plan documented

Tips & Tricks

Hard-won insights from building real agents. Read these before you start.

🧠 Prompting

  • Give the agent a persona. "You are a senior analyst" produces better output than "You are an assistant."
  • Use XML tags to structure your system prompt — Claude responds especially well to them.
  • Show examples in the prompt. One good example beats three paragraphs of instructions.
  • Version your prompts like code: commit, review, test before deploying.

🔧 Tools

  • Add a 'ask_for_clarification' tool. Letting the agent ask questions reduces errors dramatically.
  • Tool descriptions matter as much as code.
  • Log every tool call and its result. This is your primary debugging surface.

🚀 Shipping

  • Ship small, iterate fast. A scoped agent you can trust beats a powerful one you can't control.
  • Shadow mode first: run the agent in parallel with the human process before replacing it.
  • Build a feedback loop: read 50 bad outputs and ask 'why.'
💡
The #1 mistake: building a complex multi-agent system before validating the core task with a single agent and simple tools.
